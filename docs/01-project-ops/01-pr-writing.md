# PR 작성가이드

이 문서는 GitHub PR 작성 시 각 섹션별 상세 작성 방법을 안내합니다.

## PR 요구사항

### 1. 핵심 정보 누락 방지
코드리뷰나 아카이빙에 필요한 정보가 누락되지 않고 핵심적이며 간결하게 드러나야 한다.

### 2. 맥락 정보 완전성
코드변경과 관련된 맥락은 PR에 있는 정보를 통해 탐색가능해야 한다. PR에 모든 내용이 써있지 않아도 PR에 명시된 레퍼런스를 통해 찾을 수 있어야 한다.

### 3. 일관된 양식 사용
코드변경을 하는 누구나 일관된 양식을 사용하고, PR작성시 어떤 내용을 적을지에 대해 고민하지 않고, 가이드에 맞춰 내용을 추가할 수 있어야 한다.

### 4. 불필요한 정보 Optional 처리
리뷰/아카이빙에 불필요할 수 있는 정보는 optional로 기입되게 하여, PR 작성자가 억지로 내용을 맞추기 위해 불필요한 정보를 추가해야 하는 일이 없어야 한다.

### 5. 테스트 정보 필수화
코드의 변경이 테스트되었음을 보증하는 정보는 필수여야 한다. 예외적인 경우는 문서 작성, 주석 추가, 단순한 문구 수정 등이다.

## PR 제목 형식 규칙

### 제목 형식

```
{type}: {내용}
```

**Conventional Commit 형식 사용**

**핵심 규칙**:
- 타입은 필수, 콜론과 공백 한 칸 필수
- 전체 제목 50자 이하 권장 (한글 기준)
- 구체적이고 압축적인 내용 작성

### 타입 판단 로직

1. **feat**: 새로운 기능, 기능 개선
   - 사용자에게 영향을 주는 변경사항
   - 새로운 기능 구현 및 개선

2. **fix**: 버그 수정, 핫픽스
   - 오류 해결 및 버그 수정

3. **chore**: 설정 파일 변경, 빌드 관련
   - package.json, workflow, eslint 등 설정 변경
   - 의존성 업데이트, 환경 설정

4. **refactor**: 코드 리팩토링, 포맷팅
   - 기능 변경 없는 코드 품질 개선
   - **주의**: 정책 변경이나 regression 미보장시 `feat` 사용

5. **docs**: 문서 변경
   - README, 가이드, API 문서 작성/수정

6. **test**: 테스트 코드 추가/수정
   - 단위/통합 테스트, 모킹 코드 작성

7. **perf**: 성능 개선
   - 로딩 속도, 메모리, 번들 크기 최적화

### 예시

```
feat: Button 컴포넌트에 variant prop 추가
fix: 다크모드에서 텍스트 색상 미적용 수정
chore: ESLint 설정 업데이트
refactor: 결제 서비스 모듈 구조 개선
docs: API 패턴 가이드 추가
```

## PR Body 섹션별 가이드

각 PR에 공통으로 포함되는 주요 섹션들의 작성 방법을 설명합니다.

### Summary 섹션 (필수)
**목적**: 이 PR의 전체적인 목적과 내용을 요약
**작성 방법**: 목적/수단, 요구사항/구현 형태로 작성

- **목적**: 어떤 목적을 달성하기 위해 이 작업을 했는지 명확히 표현
- **수단**: 목적을 달성하기 위해 어떤 수단(코드 수정)을 활용했는지 명확히 표현

### Review Points 섹션 (Optional)
- 주요하게 리뷰가 필요한 부분을 별도로 포인트
- 리뷰어에게 이 코드를 작성하며 고민했던 부분을 설명

### Test 섹션 (필수)
**테이블 형식**: Before/After 비교
```markdown
| Before | After |
|:------:|:-----:|
| 기존 동작 | 변경된 동작 |
```

**작성 가이드**:
- 비주얼 통합 테스트: 스크린샷, 영상 등을 포함
- 재현 방법: 해당 부분을 재현할 수 있는 구체적인 방법
- 테스트 생략: 불필요한 이유와 자신감 지수를 설명

### Backlog 섹션 (Optional)
- **TODO**: 맥락분리를 위해 TODO로 남기는 부분
- **HACK**: 완벽히 이해되지 않았지만 요구사항을 달성하기 위해 만든 부분

### LLM Context 섹션 (Optional)
**사용 조건**: LLM을 활용한 작업인 경우 **반드시 작성 필수**

**작성 의도**: LLM으로 수정했을 시 **수정 결과보다는 LLM에게 요청한 내용이 리뷰에 더 중요**하기 때문

**반드시 작성해야 하는 경우**:
- 대량 코드 변환/마이그레이션 (10개 이상 파일 또는 100줄 이상)
- 복잡한 알고리즘 구현
- 도메인 특화 코드 생성

**작성하지 않아도 되는 경우**:
- 단순 수정 (오타, 변수명 변경 등)
- 기계적 작업 (복사/붙여넣기, 파일 이동 등)
- 수동 구현 (LLM 도움 없이)
- 단순 LLM 활용 (간단한 질문, 짧은 코드 조각 등)

**작성해야 할 내용**:
- 어떤 상황에서 LLM의 도움을 받았는지
- 어떤 문제를 해결하기 위해 요청했는지
- 어떤 프롬프트로 지시했는지
- 어떤 컨텍스트를 제공했는지

### References 섹션 (Optional)

**목적**: 관련 자료와 링크 제공

**포함할 수 있는 링크**:
- GitHub Issue
- 관련 Discussion / Slack Thread
- 기술 스펙, 디자인 스펙 문서
